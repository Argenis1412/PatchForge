# ADR-0003: Product Contract вЂ” Reviewable Patch Workflow

## Status

Accepted as product direction (May 2026, Updated June 2026)

## Context

The system, conceptually named **PatchForge** (implemented in this repository), began as a multi-stage agent runtime. While internal agent roles (Scout, Architect, Executor, Validator) remain useful internally for planning and execution, they are implementation details.

The core user-facing value is a safe, reviewable patch for a real repository.

## Decision

The system SHALL be developed and operated as a Git-native refactoring engine that generates, validates, and applies reviewable patches.

- **Conceptual Product Name**: PatchForge
- **Current Repository**: `PatchForge`
- **Python Package**: `orchestrator`
- **CLI Executable Name**: `patchforge` (primary), `orchestrator` (legacy alias, retained for V1 compatibility, candidate for removal in V2)

The CLI workflow remains:

```bash
patchforge doctor .
patchforge scan .
patchforge plan .
patchforge preview .
patchforge apply run_001
```

---

## Repository Safety Contract

The system SHALL NOT modify repository contents unless:

1. A patch exists.
2. Validation succeeded.
3. Repository state is compatible.
4. User explicitly executes `apply`.

This contract is the core trust boundary of the product. No action taken by the system before an explicit `apply` command may mutate the target repository's working tree.

> **вљ  Current implementation caveat**: `TargetConfig.load()` still defaults `workspace_path` to `<target>/workspace`, which means `scan` and `run` commands write logs and artifacts inside the target repository before any `apply` command. Until the workspace redesign lands, users must pass an explicit `--workspace /tmp/patchforge-workspace` (or equivalent path outside the target tree) to satisfy this contract today.

## Patch Lifecycle

Every generated patch is tied to a specific `base_commit` of the repository. Because the state of a repository can change between the time a patch is generated and when it is applied, the patch is subject to a lifecycle.

The system MUST verify the patch's validity against the current working tree state before application, resolving into one of the following states:

- **VALID**: The repository state remains compatible with `base_commit`. The patch can be applied safely.
- **STALE**: The repository state has diverged from the analyzed state. The patch must be regenerated.
- **REBASEABLE**: The repository has changed slightly, but automatic adaptation (e.g., git rebase/merge-file) of the patch is possible.
- **CONFLICT**: Incompatible changes exist. Manual intervention is required.

## Run Artifact Layout & Patch Context

Every generated patch run SHALL be self-contained and persist its complete context. No proprietary formats are allowed for human inspection.

The system SHALL generate the following artifact tree in the workspace for every run:

```text

        в”њв”Ђв”Ђ patch.diff
        в”њв”Ђв”Ђ validation.json
        в””в”Ђв”Ђ events.jsonl
```

Every generated patch run SHALL include:

- **`patch.diff`**: A standard unified diff file compatible with standard Git/patch tools.
- **`run.json`**: Run metadata (Run ID, target path, base commit, current branch, status, target goals, checksum, validation results, risk assessment, and LLM provider metadata).
- **`plan.json`**: Actionable execution steps, explaining what will change and why.
- **`validation.json`**: Detailed log of all post-generation check results (linting, test suite outputs, etc.).
- **`findings.json`**: Static repository analysis details, including detected hotspots and toolchain capabilities.
- **`events.jsonl`**: A stream of internal events recorded during execution for thorough debugging and auditability.

## Git Safety Rules

Every `apply` operation MUST be Git-safe. At a minimum, `apply` MUST:

1. Verify that the target path is a valid Git repository.
2. Record and verify the `base_commit` used to generate the patch.
3. Check whether the working tree is clean (or require an explicit override).
4. Verify that the patch resolves to a **VALID** or **REBASEABLE** state.
5. Apply the patch on an explicit, system-controlled Git branch.
6. Run post-apply validation checks.
7. Output standard Git commands (`git diff`, `git status`, `git commit`) for the developer to perform final manual review.

---

## Scope & Target Frameworks

The target capabilities of the system are intentionally bounded to maximize trust:

- **Primary Target**: Python codebases using Git, Ruff (linting), and Pytest (testing).
- **Design Intent**: Bounded refactoring tasks rather than open-ended autonomous changes.

---

## Consequences

- The patch, alongside its context, is the single unit of value.
- Runtime stages (Scout, Architect, etc.) remain hidden behind the public Scan, Plan, Preview, and Apply pipeline.
- `apply` is a distinct, explicit step representing the transition from suggestion to action.
